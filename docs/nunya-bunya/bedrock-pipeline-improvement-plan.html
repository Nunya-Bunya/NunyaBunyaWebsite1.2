Bedrock Pipeline Improvement Plan

Created: 2026-04-14

Purpose: Make Bedrock reliable enough to run end-to-end for a brand-new small business before selling it broadly.

1. Current Reality

Bedrock is viable as a founder-led done-for-you package.

It is not yet reliable as a clean one-command production pipeline.

The strongest parts are:

  • website generation
  • landing page generation
  • ad copy generation
  • lead magnet HTML/Markdown generation
  • 5-email sequence generation
  • Cal.com provisioning, with review
  • Shlink tracking links, when configured

The weakest parts are:

  • lead form handling
  • Mautic list/campaign provisioning
  • Twenty CRM setup
  • Matomo site and Metabase dashboard setup
  • polished handoff guide
  • approval gates
  • dry-run/testing mode
  • external-service preflight checks

The improvement goal is not "remove Ben from the loop."

The goal is "make Ben's loop obvious, fast, and hard to miss."

2. Test Business

Use a fake micro-business that is close enough to real client work but has no live client risk.

Recommended service test fixture:

River City Mobile Dog Wash

Type: local service business

Location: Brisbane inner north

Core offer: mobile dog wash and grooming for busy dog owners

Primary Bedrock landing-page service: first-time mobile dog wash

Why this works as a test:

  • simple local service
  • clear booking flow
  • easy lead magnet topic
  • no legal/medical/compliance complexity
  • similar enough to Pawesome without using Pawesome
  • services can be translated into booking event types

Recommended ecommerce test fixture:

Plant Riot Tees

Type: ecommerce apparel brand

Location: Australia-first, online store

Core offer: organic cotton vegan graphic t-shirt launch drop

Primary Bedrock landing-page offer: 15% first-drop discount / early access

Why this works as a test:

  • product catalog, variants, cart, checkout, and post-purchase needs expose a different class of Bedrock gaps
  • reusable for Bella Rhyder and future product brands
  • simple enough to build quickly without complex inventory rules
  • edgy brand gives the website builder a higher creative bar than a generic local service site
  • can run in waitlist-only, Stripe Payment Links, or Shopify mode

4. Required Pipeline Architecture

A. Single Run Folder

Every Bedrock run should create:

01-Nunya Bunya/Clients/[client-id]/
├── 00-admin/
│   ├── BEDROCK-RUN.md
│   ├── HANDOFF-GUIDE.md
│   ├── REVIEW-CHECKLIST.md
│   └── CLIENT-INPUT-NEEDED.md
├── 01-strategy/
├── 02-brand/
├── 03-site/
├── 04-campaigns/
├── 05-content/
├── 06-reporting/
└── BEDROCK-STATUS.json

The current orchestrator writes many outputs near the website folder.

That is workable, but the final structure needs to be predictable enough that a human can inspect it quickly.

B. Artifact Manifest

Every run should produce BEDROCK-STATUS.json with one entry per deliverable:

  • status: success, needs_review, skipped, failed
  • output path
  • public URL, if applicable
  • missing input, if any
  • external service touched
  • manual action required
  • reviewer notes

This should become the source for the handoff guide and review checklist.

C. Preflight Checks

Before generating anything, run a preflight that checks:

  • AI provider keys available
  • Vercel CLI available and logged in
  • SSH to Hetzner available, if Cal.com provisioning is enabled
  • Cal.com database reachable, if provisioning is enabled
  • Slack token present, if notifications are enabled
  • Mautic URL and credentials present
  • Twenty CRM URL and API key present
  • Matomo URL and token present
  • Shlink URL and API key present
  • required Python packages installed
  • output folder does not conflict with an active client unless --force is passed

If preflight fails, the orchestrator should tell us what will be skipped before spending tokens.

D. Run Modes

Add explicit modes:

  • --dry-run: validate config and show planned actions, no files or external services
  • --local-only: generate files only, no Vercel/Cal.com/Mautic/CRM/Slack/Matomo
  • --sandbox: generate files and use fake/sandbox external records where possible
  • --production: generate files and touch live external services

The fake business test should run first in --local-only, then --sandbox, then --production only if we deliberately want to test live integrations.

5. Improvements By Pipeline Stage

Stage 0: Client Intake Normalization

Create a canonical Bedrock input schema:

{
  "client_name": "River City Mobile Dog Wash",
  "client_id": "river-city-mobile-dog-wash",
  "industry": "mobile dog grooming",
  "location": "Brisbane inner north",
  "website_url": "",
  "primary_service": "first-time mobile dog wash",
  "services": [
    {
      "name": "First-Time Mobile Dog Wash",
      "duration_minutes": 60,
      "price_aud": 89,
      "description": "Mobile hydrobath, shampoo, dry, brush, ears, and nails."
    }
  ],
  "contact": {
    "owner_name": "Sam Taylor",
    "email": "hello@example.com",
    "phone": "0400 000 000",
    "service_area": "Brisbane inner north"
  },
  "offer": {
    "name": "First wash for $89",
    "cta": "Book the first wash"
  }
}

Required changes:

  • normalize service, primary_service, and services list into one shape
  • generate missing client_id safely
  • sanitize all client-provided strings before filesystem, SQL, HTML, or shell use
  • write normalized input to 00-admin/BEDROCK-INPUT.json

Stage 1: Website Builder

Required changes:

  • enforce a page contract: index.html, services.html, about.html, contact.html, book.html, thankyou.html
  • add optional extra pages only after the required pages pass
  • replace generic contact.html CTAs with actual booking or lead links when available
  • produce WEBSITE-QA.md with missing content, broken internal links, image count, forms found, and tracking snippets found
  • make output path match the standard client folder structure

Acceptance check:

  • all required pages exist
  • no href="#" links
  • no [MISSING] placeholders on public pages without being reported
  • all internal links point to existing files
  • mobile viewport is usable

Stage 1B: Ecommerce Website Builder

Required changes:

  • add business_model: ecommerce as a first-class branch
  • generate product catalog data separately from the rendered site
  • support collection pages, product cards, product detail content, cart state, discount capture, and checkout CTA
  • support checkout modes: waitlistonly, stripepaymentlinks, shopify, externalcheckout
  • generate ecommerce-specific policy placeholders: shipping, returns, privacy, terms
  • generate product image briefs and placeholder mockups when no real product images exist
  • produce ECOMMERCE-QA.md with product count, missing variants, missing checkout URLs, missing policies, and event tracking status

Acceptance check:

  • every product has ID, title, price, description, image, variants, and checkout behavior
  • cart or product checkout path is present
  • signup path is present
  • policy gaps are listed in CLIENT-INPUT-NEEDED.md
  • no live store can launch with placeholder checkout links unreported

Stage 2: Landing Page And Forms

Required changes:

  • stop using an unverified generic /api/leads action
  • create a Vercel/serverless form handler or static form backend per client
  • submit lead to Mautic, Twenty CRM, and Slack
  • write leads locally/Supabase as a fallback
  • add hidden fields: clientid, offerslug, source, campaign
  • create thankyou.html with booking CTA and lead magnet delivery CTA

Acceptance check:

  • test form submission creates a lead record
  • Slack receives a lead alert
  • Mautic receives the contact or the run clearly says Mautic was skipped
  • CRM receives the person/deal or the run clearly says CRM was skipped

Stage 3: Lead Magnet

Required changes:

  • add automated HTML to PDF export
  • save both guide.html and guide.pdf
  • add a lead magnet download link to the thank-you page
  • include disclaimer blocks for regulated industries
  • add quality check for title, section count, CTA, and client name

Acceptance check:

  • guide.pdf exists
  • PDF opens
  • guide has at least 6 substantive sections
  • guide includes client name and CTA

Stage 4: Email Sequence And Mautic

Required changes:

  • generate email files as now
  • create Mautic segment per client
  • create Mautic form or connect existing form endpoint
  • create Mautic email drafts
  • create or document a campaign sequence
  • remove hardcoded Zinia-specific directions from email prompts
  • write EMAIL-SETUP.md with IDs, status, and manual UI actions

Acceptance check:

  • segment ID saved
  • all 5 emails created as drafts
  • test contact can be added to segment
  • no fake address, no Hakataya/Ramen/Zinia leftover text

Stage 5: Booking System

Required changes:

  • replace raw SQL string interpolation with parameterized or safely escaped SQL helpers
  • make Cal.com provisioning idempotent
  • detect existing user, schedule, event types, and webhooks
  • support --local-only by generating BOOKING-SETUP.md without touching Cal.com
  • generate a booking QA report listing services, durations, prices, and booking URL

Acceptance check:

  • booking URL resolves
  • expected event types exist
  • no duplicate schedules/event types on rerun
  • webhook points to the correct n8n endpoint

Stage 6: Slack And Notifications

Required changes:

  • create or find the client Slack channel idempotently
  • post a Bedrock run summary into the channel
  • verify n8n booking webhook receiver is live
  • route Cal.com booking events to the client channel
  • add a test notification command

Acceptance check:

  • Slack channel exists or is skipped with a clear reason
  • test lead/booking notification posts to the right channel

Stage 7: CRM

Required changes:

  • implement stepcrmentry
  • create company
  • create primary contact
  • create opportunity/deal for Bedrock
  • write CRM IDs to BEDROCK-STATUS.json

Acceptance check:

  • Twenty CRM has company/contact/opportunity or a clear skipped status

Stage 8: Analytics And Tracking

Required changes:

  • create Matomo site and save site ID
  • inject Matomo tracking snippet into website and landing page
  • create goals for form submit, booking click, thank-you visit
  • create Shlink links for site, landing page, booking, lead magnet
  • create or clone a basic Metabase dashboard template

Acceptance check:

  • Matomo site ID saved
  • tracking snippet present on generated pages
  • Shlink links resolve
  • Metabase dashboard created or skipped with clear reason

Stage 9: Ads And Launch Assets

Required changes:

  • keep current ad generation
  • add campaign brief file with offer, audience, landing page URL, tracking links
  • add "do not publish until reviewed" status
  • produce a lightweight launch checklist

Acceptance check:

  • Meta and Google variations exist
  • tracking URLs are included
  • launch checklist exists

Stage 10: Handoff And Review

Required changes:

  • generate REVIEW-CHECKLIST.md
  • generate HANDOFF-GUIDE.md
  • generate CLIENT-INPUT-NEEDED.md
  • collect every public URL and admin URL in one place
  • separate "ready for client" from "needs Ben review"

Acceptance check:

  • Ben can open one folder and know exactly what is done, skipped, failed, and ready
  • client handoff does not expose internal secrets

6. Recommended Build Order

Pass 1: Make It Testable

  1. Add canonical input schema and fake test fixture.
  2. Add --dry-run and --local-only.
  3. Add preflight checks.
  4. Add artifact manifest helpers.
  5. Add final review/handoff file generation.

Why first:

Without this, every test run is noisy and hard to trust.

Pass 2: Fix The Critical Lead Path

  1. Replace /api/leads with a working lead endpoint or form backend.
  2. Push lead to Supabase/local log as fallback.
  3. Push lead to Slack.
  4. Push lead to Mautic if configured.
  5. Push lead to Twenty CRM if configured.

Why second:

This is the most commercially important path.

Pass 3: Wire Marketing Stack Integrations

  1. Mautic segment creation.
  2. Mautic email draft creation.
  3. Twenty CRM company/contact/deal.
  4. Matomo site creation and snippet injection.
  5. Shlink links for all public URLs.
  6. Metabase dashboard placeholder or clone.

Why third:

These are the things that make Bedrock feel like a real operating system, not just generated assets.

Pass 4: Harden Booking

  1. Cal.com idempotency.
  2. Safer SQL handling.
  3. Booking event QA.
  4. n8n webhook verification.
  5. Slack booking notification test.

Why fourth:

Booking is valuable, but less dangerous than broken lead capture because it is easier to see manually.

Pass 5: Polish Generated Assets

  1. PDF export.
  2. better landing page templates.
  3. stronger lead magnet templates by industry.
  4. better ad/creative brief.
  5. screenshot QA for generated website/landing pages.

Why fifth:

Quality matters, but structure comes before shine.

7. Fake Business End-To-End Test Plan

Test 1: Local-Only Generation

Run the fake business through the pipeline without touching external services.

Expected:

  • website generated
  • landing page generated
  • lead magnet HTML/Markdown/PDF generated
  • email sequence generated
  • ad campaign generated
  • handoff and review docs generated
  • external integration statuses are skippedlocalonly

Test 2: Sandbox Integration

Run the fake business with safe external services enabled.

Expected:

  • test lead submits successfully
  • Slack receives test notification
  • Mautic receives test contact
  • CRM receives test company/contact/deal
  • Shlink links created
  • Matomo snippet present

Test 3: Rerun Idempotency

Run the same fake business again.

Expected:

  • no duplicate Cal.com users/event types
  • no duplicate Slack channels
  • existing folders are reused or versioned intentionally
  • status file reports existing resources

Test 4: Human Review

Open REVIEW-CHECKLIST.md and inspect every client-facing item.

Expected:

  • no unsupported claims
  • no placeholder content
  • no wrong industry details
  • no broken links
  • no generated copy that sounds fake or legally risky

8. Ship Criteria Before Advertising Hard

Bedrock is ready to advertise aggressively when:

  • a fake business can run end-to-end twice without uncontrolled duplication
  • one test lead flows from landing page to Slack and CRM
  • Mautic list and email drafts are created or clearly marked manual
  • every deliverable appears in the final handoff guide
  • the review checklist catches missing data and placeholders
  • the package can be completed in one workday with Ben reviewing, not manually assembling

Until then, advertise softly as founder-led implementation slots.

9. Non-Negotiable Principle

Do not optimize for "fully automated."

Optimize for:

  • complete
  • reviewable
  • rerunnable
  • safe
  • useful to a real business owner

Automation is the engine.

The product is the finished marketing foundation.
