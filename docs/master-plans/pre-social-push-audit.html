Pre-Social Push Audit — Site Readiness Checklist

🚀 Master Plans MASTER PLANS

Pre-Social Push Audit — Site Readiness Checklist

Use this every time you're about to push a business on social and want to make sure the site can actually convert.

Last updated: 2026-04-20

PHASE 1 — CODE AUDIT (Claude handles this)

Run through these for every site before a social push. These are the things that silently kill conversions without anyone noticing.

1. Booking Links

  • ☐ Every "Book Now" / "Get Started" / "Book a Session" CTA has a real URL (not #, not javascript:void)
  • ☐ All booking links go to a live Cal.com event (or equivalent)
  • ☐ Nav CTA goes directly to booking, not to a scroll anchor that adds friction
  • ☐ Booking links open in a new tab (target="_blank")

2. Lead Capture Forms

  • ☐ Every form has a submit handler (no orphaned <form> tags with no JS)
  • ☐ Forms POST to a real API endpoint (check the fetch() URL)
  • ☐ Required fields are validated before submit
  • ☐ Button disables during submission (prevents double-submits)
  • ☐ Success state is handled (redirect to thank you page OR clear inline message)
  • ☐ Error state is handled (button re-enables, error shown)

3. Lead Routing — Mautic

  • ☐ Form submissions create/update a Mautic contact via the contacts API (/api/contacts/new) — NOT the form submission endpoint (/form/submit) which requires MAUTICFORMID
  • ☐ Contact is added to the correct segment after creation
  • ☐ Tags are applied (['business-name', 'source'])
  • ☐ The correct env vars are available: MAUTICURL, MAUTICAPIUSER / MAUTICUSER, MAUTICAPIPASSWORD / MAUTICPASS, MAUTIC*SEGMENTID

4. Lead Routing — Slack

  • ☐ Contact form triggers a Slack notification with name, email, phone, message
  • ☐ Lead magnet signup triggers a Slack notification
  • ☐ Notifications go to the correct channel ID (check slack_workspace.md for IDs)
  • ☐ SLACKBOTTOKEN env var is set in Vercel

5. Lead Routing — Supabase (NB only)

  • ☐ Form submissions insert a row into the leads table
  • ☐ SUPABASEURL and SUPABASEKEY env vars are set
  • ☐ Required columns exist in the table (name, email, phone, source, page, interest, message, created_at)

6. Lead Magnet Email Delivery

  • ☐ The subscribe/lead magnet API sends a Mautic email to the new contact
  • ☐ The email ID env var is set correctly (e.g. MAUTICPPLEADMAGNETEMAILID=85)
  • ☐ The email ID default in code matches the actual welcome/Day 0 email in Mautic
  • ☐ The lead magnet PDF/asset is actually linked in that email (not a placeholder)

7. Analytics & Tracking

  • ☐ Meta Pixel is present on every page (not just landing pages)
  • ☐ Pixel ID is correct (matches the actual Meta Business account)
  • ☐ Matomo / Google Analytics script is present
  • ☐ Schema.org structured data exists (LocalBusiness type at minimum)

8. Thank You Pages

  • ☐ /thankyou.html or /thankyou exists and loads
  • ☐ Thank you page has a next step (book a call, follow on socials, share)
  • ☐ Lead magnet forms redirect to thank you page after submit
  • ☐ Contact forms either redirect or show a clear inline success message

9. Vercel Config

  • ☐ vercel.json exists with cleanUrls: true and API rewrites
  • ☐ All required env vars are listed (compare process.env.* usage in /api/ against actual Vercel dashboard)
  • ☐ No API keys hardcoded in HTML or JS files (should all be process.env.* in serverless functions)
  • ☐ Redeploy after any env var changes — changes don't go live until you trigger a new deployment

10. Social Links

  • ☐ Every social icon in nav/footer goes to a real, live profile URL
  • ☐ Links open in a new tab
  • ☐ No placeholder # links remaining

11. Mobile

  • ☐ Open on actual phone (not just Chrome dev tools)
  • ☐ Nav collapses correctly
  • ☐ CTAs are tappable (not too small, not overlapping)
  • ☐ Text doesn't overflow or get cut off
  • ☐ Forms are usable on mobile keyboard

12. Broken Links

  • ☐ All internal page links go to real pages
  • ☐ No 404s on package, service, or portfolio pages
  • ☐ Blog exists and has at least one real post (or is removed from nav)

PHASE 2 — MANUAL CHECKS (Ben does these — ~1.5 hrs)

Cal.com

  • ☐ All booking events exist in the Cal.com dashboard
  • ☐ Events are set to Active (not draft/hidden)
  • ☐ Each event has a confirmation email set for the booker
  • ☐ Each event has a notification email going to you
  • ☐ Redirect URL after booking is set (thank you page or Cal.com default success screen)
  • ☐ Event durations and prices match what's on the website

Vercel Env Vars

  • ☐ Go to Vercel → Project → Settings → Environment Variables
  • ☐ Compare against the list of process.env.* calls in /api/ files
  • ☐ All required vars are present for Production environment
  • ☐ No vars are blank or set to placeholder values

Mautic Campaigns

  • ☐ Log into mautic.nunyabunya.com
  • ☐ Each email nurture sequence is Published (not Draft)
  • ☐ Day 0 welcome email has real content (not "Hello [firstname]" placeholder)
  • ☐ Lead magnet asset is linked in the delivery email (PDF, Google Drive, etc.)
  • ☐ Campaign is set to trigger when contact is added to segment

End-to-End Test

  • ☐ Submit a real test form using your own email
  • ☐ Slack notification fires within 30 seconds
  • ☐ Mautic contact appears in the correct segment
  • ☐ Welcome/guide email arrives in inbox within 5 minutes
  • ☐ Book a fake test appointment via Cal.com
  • ☐ Confirmation email arrives in inbox within 2 minutes

Social Links Live Check

  • ☐ Click every social icon on the live site (not local)
  • ☐ Each goes to the correct profile
  • ☐ Profiles are active (not empty, not suspended)
