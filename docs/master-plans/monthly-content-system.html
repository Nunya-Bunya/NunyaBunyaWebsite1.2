Monthly Content Generation System

🚀 Master Plans MASTER PLANS

Monthly Content Generation System

Goal: One command generates a full month of content for any client. The human role becomes assembly and curation (~2.5 hours per client per month).

Operating Principle

Generate everything first. Curate second. Publish third. Learn fourth.

This system is designed to make monthly marketing execution fast, consistent, testable, and highly scalable across multiple clients. The operator is not the writer. The operator is the creative director.

Core Outcome

For each client, the pipeline reliably produces a month of marketing inventory:

  • 80-100 caption drafts (with hook variations)
  • 4-8 carousel drafts
  • 8-12 short-form video scripts
  • 27+ ad copy variations
  • 20-30 story templates
  • 4 email campaigns
  • visual pairings for all approved posts
  • a filled monthly content calendar
  • a surplus/swap pool of backup content
  • a video shoot list for batch filming day
  • a feedback loop that improves next month's generation

The human operator curates the month in roughly 2.5 hours.

1. Raw Material Input Layer

These inputs must exist before any generation runs. The quality of inputs directly governs whether outputs feel like the client or like generic AI content.

1.1 Brand Bible

Structured brand source of truth. Stored in pipelinedocuments as doctype brand_bible.

Required fields:

  • brand name
  • one-line positioning statement
  • audience definition
  • tone / voice traits
  • words to use
  • words to avoid
  • offer(s) and CTA library
  • visual system (colors, fonts, visual references)
  • content do/don't rules
  • platform priorities

Already built: 21-agent Brand Bible pipeline produces this. Existing client Brand Bibles in pipeline_documents.

1.2 Content Pillars

Recurring categories the brand can speak about forever. 4-6 pillars per client.

Required data per pillar:

  • pillar name
  • pillar description
  • examples of acceptable topics
  • examples of unacceptable topics
  • preferred CTA types
  • best format types for this pillar (static, carousel, reel, etc.)

Currently stored in: clientconfigs.json as contentpillars (name only). Gap: needs to be expanded to include per-pillar metadata (acceptable/unacceptable topics, preferred CTAs, best formats). This becomes a new field contentpillardetails in client config or a dedicated content_pillars table.

1.3 Swipe File

Structured inspiration library. These are not copied — they are decomposed into reusable structure.

Store each entry as:

  • source URL or screenshot
  • platform
  • niche
  • hook type
  • angle
  • format
  • CTA type
  • why it works (structural breakdown)
  • reusability score (1-5)

Function: Used as structural prompt fuel for hooks, content flow, and creative variation. Fed into the monthly planner and caption factory as templates.

Currently stored in: nb-admin-swipe-file.html exists as a UI page. Gap: needs a Supabase swipe_entries table and an ingestion flow.

1.4 Asset Library

Approved visual material the system can use automatically.

Includes:

  • AI brand photos
  • real client photos
  • product images
  • logos and icons
  • branded design templates (quote cards, stat graphics, carousels, story templates, ad templates)
  • background textures
  • B-roll clips

Required metadata per asset:

  • asset type
  • aspect ratio
  • campaign or pillar fit
  • usage rights / approval status
  • tags (searchable)
  • last used date (for freshness rotation)

Currently stored in: /Content Bank/ with 6 organized folders (00-ready-to-use through 99-review-or-delete). Gap: no metadata database. Assets are organized by folder but not queryable by tag, pillar, or aspect ratio. Needs a Supabase assets table or metadata sidecar files.

1.5 Offer Library

Centralized list of the things the business wants people to do. Prevents content from floating aimlessly without commercial purpose.

Includes:

  • core offer (the main thing you sell)
  • seasonal offers
  • urgency offers (limited time, limited spots)
  • lead magnets (free downloads, audits, consultations)
  • low-friction CTAs (follow, save, share, comment)
  • booking CTAs (calendar links, DM triggers)

Currently stored in: partially in offerstrategy pipeline document, partially in clientconfigs.json as cta_link. Gap: needs to be a structured list per client, queryable by type.

1.6 Input Normalizer Agent

Purpose: Takes raw client materials (messy brand docs, scattered assets, verbal instructions from onboarding) and converts them into the structured format the pipeline expects.

File: Execution/input_normalizer.py

Directive: Directives/input_normalizer.md

Registry: "inputnormalizer": ("inputnormalizer", "runinputnormalizer")

When to run: Once per new client during onboarding, or when client materials change significantly.

Input: Raw brand docs, URLs, uploaded files, survey responses (from BFK/Brand Bible pipeline)

Output: Validated, structured entries in all input tables (pillars with metadata, offers, asset metadata, initial swipe entries)

3. Pipeline Definition

Goes in deliverablepipelineorchestrator.py alongside existing pipelines. Maps to the existing PipelineStep / PipelineDefinition dataclass pattern.

MVP Pipeline (V1)

PIPELINE_DEFINITIONS["monthly_content_v1"] = PipelineDefinition(
    name="monthly_content_v1",
    description="MVP monthly content generation — captions + reel scripts + simple curation. Visual pairing and calendar assembly are manual.",
    steps=[
        # Step 1: Plan the month
        PipelineStep(
            agent_name="monthly_content_planner",
            output_key="monthly_plan",
            input_from=["brand_voice", "audience_research"],
        ),

        # Step 2-3: Bulk generation (parallel)
        PipelineStep(
            agent_name="bulk_caption_generator",
            output_key="caption_drafts",
            input_from=["monthly_plan", "brand_voice"],
            parallel_group=10,
        ),
        PipelineStep(
            agent_name="bulk_reel_scripts",
            output_key="reel_script_drafts",
            input_from=["monthly_plan", "brand_voice"],
            parallel_group=10,
        ),

        # Step 4: REVIEW GATE — curate in approval queue
        PipelineStep(
            agent_name="monthly_content_planner",  # no-op, gate only
            output_key="review_checkpoint",
            requires_review=True,
        ),
    ]
)

Full Pipeline (V3)

PIPELINE_DEFINITIONS["monthly_content_gen"] = PipelineDefinition(
    name="monthly_content_gen",
    description="Full monthly content generation — all content types, visual pairing, calendar assembly, platform adaptation, and scheduling.",
    steps=[
        # Step 1: Plan the month — pillars, themes, topic distribution
        PipelineStep(
            agent_name="monthly_content_planner",
            output_key="monthly_plan",
            input_from=["brand_voice", "audience_research", "competitor_analysis"],
        ),

        # Step 2: Bulk generation (all parallel — independent of each other)
        PipelineStep(
            agent_name="bulk_caption_generator",
            output_key="caption_drafts",
            input_from=["monthly_plan", "brand_voice"],
            parallel_group=10,
        ),
        PipelineStep(
            agent_name="bulk_carousel_generator",
            output_key="carousel_drafts",
            input_from=["monthly_plan", "brand_voice"],
            parallel_group=10,
        ),
        PipelineStep(
            agent_name="bulk_reel_scripts",
            output_key="reel_script_drafts",
            input_from=["monthly_plan", "brand_voice"],
            parallel_group=10,
        ),
        PipelineStep(
            agent_name="bulk_ad_variations",
            output_key="ad_variation_drafts",
            input_from=["monthly_plan", "brand_voice", "offer_strategy"],
            parallel_group=10,
        ),
        PipelineStep(
            agent_name="bulk_story_templates",
            output_key="story_drafts",
            input_from=["monthly_plan", "brand_voice"],
            parallel_group=10,
        ),
        PipelineStep(
            agent_name="bulk_email_sequence",
            output_key="email_drafts",
            input_from=["monthly_plan", "brand_voice"],
            parallel_group=10,
        ),

        # Step 3: REVIEW GATE — curate all generated content
        PipelineStep(
            agent_name="monthly_content_planner",  # no-op, gate only
            output_key="curation_checkpoint",
            requires_review=True,
        ),

        # Step 4: Visual pairing (only pairs approved content — saves API calls)
        PipelineStep(
            agent_name="visual_pairing_engine",
            output_key="visual_pairings",
            input_from=["caption_drafts", "carousel_drafts", "ad_variation_drafts"],
        ),

        # Step 5: Assemble calendar from approved content
        PipelineStep(
            agent_name="monthly_calendar_assembler",
            output_key="draft_calendar",
            input_from=[
                "monthly_plan",
                "caption_drafts",
                "carousel_drafts",
                "reel_script_drafts",
                "ad_variation_drafts",
                "story_drafts",
                "email_drafts",
                "visual_pairings",
            ],
        ),

        # Step 6: Generate platform-specific variants
        PipelineStep(
            agent_name="platform_adapter",
            output_key="platform_variants",
            input_from=["draft_calendar"],
        ),

        # Step 7: Final review of calendar before scheduling
        PipelineStep(
            agent_name="monthly_calendar_assembler",  # no-op, gate only
            output_key="scheduling_checkpoint",
            requires_review=True,
        ),

        # Step 8: Push to scheduling platforms
        PipelineStep(
            agent_name="schedule_approved_content",
            output_key="scheduling_result",
            input_from=["draft_calendar", "platform_variants"],
        ),
    ]
)

Key difference between V1 and V3: In V1, visual pairing is manual (human picks images during curation). In V3, the review gate comes BEFORE visual pairing — so the system only fetches/generates images for approved content, saving API calls and compute.

5. Curation Layer

5.2 Curation Rubric

Every generated asset is judged by:

  1. Brand fit — does it sound like this brand?
  2. Clarity — is the message immediately understandable?
  3. Novelty — is this saying something the feed hasn't said recently?
  4. Relevance — does this connect to a current goal or offer?
  5. Engagement likelihood — would this stop a scroll?
  6. Conversion likelihood — does this move someone closer to action?
  7. Non-cringe factor — would you be proud to post this?
  8. Platform fit — is this native to the platform it's for?

Statuses:

  • Keep — approve as-is
  • Tweak — inline edit, then approve
  • Cut — reject, moves to surplus pool (not deleted — available for swaps)

10. Success Criteria

The system works if:

  • A month of content can be generated in one pipeline run (~15 min execution)
  • Curation stays under 2.5 hours per client
  • Output quality stays brand-aligned (passes the curation rubric)
  • Posting cadence is consistent (no dead zones, no burst-and-disappear)
  • Testing variation exists by default (every topic gets multiple hooks)
  • Monthly learnings influence the next generation cycle
  • Adding a new client means running the input normalizer, not rebuilding the pipeline
  • The operator feels like a creative director, not a content hamster

12. Human Role Definition

The operator is not the writer. The operator is the creative director.

Human responsibilities:

  • Define strategy inputs correctly (brand bible, pillars, offers)
  • Spot weak or off-brand outputs during curation
  • Approve the strongest assets
  • Decide filming priorities for video content
  • Approve ad test budgets
  • Protect taste and brand consistency
  • Interpret metrics at a strategic level
  • Make judgment calls the system can't (timing, sensitivity, cultural context)

Human should NOT spend time on:

  • Writing every post from scratch
  • Resizing assets by hand
  • Inventing fresh content ideas every day
  • Calendar improvisation
  • Manually pulling metrics from each platform
  • Reformatting content per platform
